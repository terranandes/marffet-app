## Summary

This review covers the 4-bug fix chain that resolved the Data Integrity Crisis (2000-2004 backfill silently producing 0 rows).

## Changes Reviewed

### 1. `market_data_service.py` — MultiIndex Fix (L861-864)

```diff
+df_ticker = data
+if isinstance(data.columns, pd.MultiIndex) and ticker in data.columns:
+    df_ticker = data[ticker]
```

**Verdict**: ✅ APPROVED.
- Correctly handles `yfinance` 1.0+ behavior where `MultiIndex` is returned even for single-ticker downloads.
- Without this, `row['Open']` silently fails → 0 rows accumulated.

### 2. `market_data_service.py` — SAVE_INTERVAL Dedup (L832)

```diff
-SAVE_INTERVAL = 10 if IS_CLOUD else 200
+SAVE_INTERVAL = 10 if IS_CLOUD else 20
```

**Verdict**: ✅ APPROVED.
- A second assignment at L832 was overriding the one at L755. Both now consistently use 20.
- 200 stocks × ~1000 rows = 200K row flush caused DuckDB `executemany` hangs.

**⚠️ Recommendation**: Remove the duplicate at L755 entirely. Single source of truth.

### 3. `backfill_2000_2004_resumable.py` — Connection Close (L35-38)

```diff
+finally:
+    conn.close()
```

**Verdict**: ✅ APPROVED.
- `read_only=True` connection was never closed, preventing `read_only=False` `flush_results` from acquiring a lock.
- DuckDB does not support concurrent R/O + R/W connections locally.

### 4. `supplement_splits.py` — Schema Fix (L58-63)

```diff
-INSERT INTO dividends (stock_id, year, cash, stock, stock_id_year) VALUES (?, ?, ?, ?, ?)
+INSERT INTO dividends (stock_id, year, cash, stock) VALUES (?, ?, ?, ?)
```

**Verdict**: ✅ APPROVED.
- `stock_id_year` column does not exist. PK is composite `(stock_id, year)`.

## Open Items

1. **L755 SAVE_INTERVAL**: Redundant. Should be removed to prevent future confusion.
2. **Debug prints**: Multiple `[DEBUG]` prints remain in `market_data_service.py`. Should be gated behind a `DEBUG` flag or removed before production.
3. **Exception handling**: Lines 909 (`except: continue`) and 919 (`except: pass`) are bare `except` clauses. These should be `except Exception` at minimum and ideally log the error.

## Risk Assessment

- **Low Risk**: All changes are additive or corrective.
- **Data Safety**: `ON CONFLICT DO UPDATE` ensures no duplicate rows.
- **Rollback**: If needed, the DuckDB can be rebuilt from MI_INDEX Parquet snapshots.
